# 4. 规范文档内容结构

### 4.1 规范驱动开发原则
即实行 **文档驱动开发**：规范文档不仅是开发的指导，更是代码的直接依据，从而彻底告别"开发完成后补写文档"的低效模式。
秉承 spec-kit 的核心理念：**让规范可执行**——需求文档不应只是静态的约束，而应直接转化为可用的代码实现。
你需要严格遵照 `SpecMd` 目录下的文档进行开发，确保每一项需求变更都能直接推动代码的演进。
具体执行标准（**由宏观到微观的层级化规范**）：
**项目规范和需求文档文件夹路径(Level 1 和 Level 2)固定为 `./SpecMd`**
**文件规范(Level 3)固定为每个代码文件的开头，用三引号包裹，称为文档字符串（docstring）**

**需求迭代** (`Specify.md`): 唯一的需求变更源与版本控制中心。任何需求变化必须先更新此文档，再推动代码修改,做好需求文档版本控制, **明确标注需求变化**，更新需求状态，比如从未实现变成已实现未测试。

- **Level 1: 宏观架构与全局控制 (Global Scope)**
    - **项目架构** (`Structure.md`): 包含项目目标、架构设计、技术栈及模块职责。整体代码结构必须与之 **一一对应** 并 **同步更新**。
    - **通用准则** (`ProjectRules.md`): 全局适用的代码编写规则与底线。

- **Level 2: 模块协作与层级规范 (Module & Layer Scope)**
    - **流程与交互** (`/SpecMd/workflows/`): 定义工作流及数据流转规范(如果有)。模块间的交互逻辑必须与之 **一一对应** 并 **同步更新**。
    - **层级实现** (`/SpecMd/layers/`): 针对特定代码层（如数据层、模型层）的详细说明文档。各层的代码实现必须与之 **一一对应** 并 **同步更新**。
    - **智能体检查报告** (`/SpecMd/agents/`): 存放各智能体的检查结果和状态报告。
    - **其他规范** (`/SpecMd/others/`): 依据项目变化而变化的特殊规范文档。

- **Level 3: 微观文件实现 (File Scope)**
    - **文件自描述** (docstring): 包含文件的功能（涵盖公共类及核心函数，忽略私有/辅助细节）即为该文件的“需求文档”，该文件的代码实现必须与之 **严格对应** 并 **同步更新**。

### 4.2 层级文档格式要求 (`SpecMd/layers/`)
- 该层的**目的**和适用范围（必填）
- 验证**规则**和约束条件（必填），例如详细说明**数据ID与样本ID（数据掩码）的一致性要求**
- 与其他层的**集成**要点（必填）
- 该层的代码或文件**结构**（必填）

### 4.3 需求文档内容结构 (`Specify.md`)

需求文档是项目的核心驱动力，必须包含以下标准化字段：

1.  **任务标题 (Task Title)**:
    -   格式: `# [任务名称] 任务`
    -   描述: 简洁明了地说明当前迭代的核心目标。

2.  **需求实施路线图 (Step-by-Step Implementation Roadmap)**:
    -   格式: 分阶段的 Checkbox 列表 (`- [ ]`)。
    -   内容: 将大任务拆解为可执行的微小步骤（如“基础架构”、“接口定义”、“具体实现”、“清理与验证”）。
    -   作用: 追踪任务进度，指导开发顺序。

3.  **需求与任务细化 (Detailed Requirements)**:
    -   内容: 对路线图中的每个步骤进行详细的技术说明。
    -   **关键字段**:
        -   **功能描述**: 具体的业务逻辑或功能点。
        -   **数据结构**: 输入输出的数据类型、维度、格式要求。
        -   **验证规则**: 如何判断该功能实现正确（如“输入维度必须为(N,2)”）。
        -   **接口定义**: 函数签名、参数说明、返回值说明。
        -   **分析场景**: 该功能解决的具体问题或应用场景。

4.  **复用组件与集成说明 (Integration & Reusability)**:
    -   内容: 说明新功能如何与现有系统集成，以及提取的公共组件。

5.  **验证标准 (Verification Criteria)**:
    -   内容: 明确的验收标准（如“通过某某测试脚本”、“生成某某文件”、“Docstring包含特定内容”）。

6.  **任务迭代历史 (Iteration History)**:
    -   格式: 时间戳 + 版本号 + 变更摘要。
    -   内容: 记录需求文档的变更历史，确保可追溯性。

### 4.4 流程与交互文档内容结构 (`SpecMd/workflows/`)
用于定义跨模块的复杂交互逻辑或数据流转过程。

- **流程名称 (Workflow Name)**: 清晰标识该流程的业务含义。
- **参与者 (Participants)**: 列出涉及的模块、类或外部系统。
- **触发条件 (Trigger Conditions)**: 流程开始的前置条件或事件。
- **步骤详解 (Step-by-Step Flow)**: 
    - 必须包含**数据流向** (Input -> Process -> Output)。
    - 关键步骤需引用具体的代码层或接口。
- **异常处理 (Exception Handling)**: 定义流程中可能出现的错误及系统的预期行为。
- **时序图/流程图 (Optional)**: 推荐使用 Mermaid 语法描述复杂逻辑。

### 4.5 项目架构文档内容结构 (`Structure.md`)
作为项目的全局地图，必须保持实时更新。

- **项目目标 (Project Goal)**: 一句话描述项目的核心价值和功能。
- **系统架构 (System Architecture)**: 
    - 定义核心模块及其职责边界。
    - 说明模块间的依赖关系。
- **目录映射 (Directory Mapping)**: 
    - 关键目录/文件与架构模块的对应关系。
    - 必须与实际文件系统保持一致。
- **技术栈 (Tech Stack)**: 列出核心依赖库及其版本要求。

### 4.6 通用准则文档内容结构 (`ProjectRules.md`)
定义项目的底线规则。

- **代码风格 (Coding Style)**: 命名规范、注释要求、格式化标准。
- **环境规范 (Environment)**: 操作系统、Python 版本、包管理工具 (my-uv) 的使用说明。
- **开发流程 (Development Workflow)**: 从需求分析到代码提交的标准步骤。
- **错误处理原则 (Error Handling)**: 如“默认不使用 try-except”等核心原则。

### 4.7 智能体检查报告文档内容结构 (`SpecMd/agents/`)
用于记录智能体自动执行任务的结果。

- **检查元数据**:
    - **检查者**: 执行任务的智能体名称。
    - **时间戳**: 报告生成时间。
    - **检查对象**: 被检查的代码文件或文档范围。
- **检查项列表 (Checklist)**:
    - 使用 `[ ]` / `[x]` 标记每个检查点的状态。
- **发现的问题 (Issues Found)**:
    - 详细描述发现的不一致、错误或潜在风险。
    - 必须引用具体的文件和行号。
- **改进建议 (Action Items)**:
    - 针对发现的问题提出的具体修复建议。
