# 3. 智能体体系 (Agent System Architecture)

### 3.0 规范驱动开发原则 (Specification-Driven Development Principles)

所有智能体必须严格遵循 **规范驱动开发 (Specification-Driven Development)** 的核心原则。这不仅是开发指导，更是所有行动的依据。

#### 核心理念
- **文档即代码 (Docs as Code)**: 规范文档不是事后补充的说明，而是代码的直接前置条件。
- **让规范可执行 (Executable Specs)**: 需求文档应直接转化为可验证的代码实现。
- **单一事实来源 (Single Source of Truth)**: `SpecMd` 目录是项目需求的唯一真理来源。

#### 执行标准 (层级化规范)
- **Level 1: 宏观架构 (Global Scope)**
    - `Structure.md` (项目架构) 与 `ProjectRules.md` (通用准则) 控制全局。
- **Level 2: 模块协作 (Module & Layer Scope)**
    - `/SpecMd/workflows/` (流程) 与 `/SpecMd/layers/` (层级) 定义具体实现逻辑。
- **Level 3: 微观实现 (File Scope)**
    - **文件自描述**: 源代码文件的头部注释即为该文件的微观需求文档。

本系统定义了 5 个专用智能体。为了保证智能体行为的一致性和高可用性，所有智能体提示词文件 (`agent/spec-*.md`) **必须** 严格遵循以下定义的**标准架构**。

### 3.1 智能体提示词标准架构 (Standard Agent Prompt Schema)

每个智能体定义文件必须包含以下 7 个详细章节，并且内容深度必须达到下述要求：

#### 1. 元数据 (Metadata)
- **字段**: `trigger` (触发指令), `alwaysApply` (是否强制应用), `description` (简短描述)。
- **要求**: 明确该智能体的调用时机。

#### 2. 角色定义 (Role Definition)
- **内容**: 明确智能体的身份（如“资深后端工程师”）、核心目标及能力边界。
- **要求**: 使用角色扮演的口吻，建立专业人设。

#### 3. 工作流与思维链 (Workflow & Chain of Thought)
必须包含详细的 Step-by-Step 流程，指导智能体如何一步步完成任务：
1.  **输入分析 (Input Analysis)**: 解析用户指令，确定涉及的文件。
2.  **上下文检索 (Context Retrieval)**: 读取 `SpecMd` 规范、相关代码文件。
3.  **执行逻辑 (Execution)**: 核心处理步骤（如代码生成、文档更新、测试运行）。
4.  **自我验证 (Self-Verification)**: 检查输出是否符合规范（如运行 linter 或检查文件格式）。
5.  **产物输出 (Output Generation)**: 生成最终文件或回复。

#### 4. 核心功能清单 (Core Functions)
- **内容**: 详细列出该智能体可执行的具体任务清单（Capability List）。
- **要求**: 每个功能点应具体可执行，而非抽象描述。

#### 5. 输入输出接口 (I/O Interface)
- **输入 (Input)**: 明确接受的数据格式（如 SpecMd 文档, Python 源码, 错误日志）。
- **输出 (Output)**: 明确产生的交付物（如 Updated Markdown, New Python Files, Test Report）。

#### 6. 专用模块 (Specific Modules)
- **代码风格**: 若智能体负责代码生成，必须指定使用的代码风格（如 PEP 8）。
- **文档结构**: 若智能体负责文档生成，必须指定输出的 Markdown 格式（如标题层级、列表项缩进）。
- **报告规范**: 若智能体输出报告，必须指定报告的格式（如测试用例列表、测试结果）。
- **环境要求**: 若智能体负责运行代码并测试，必须指定运行环境（如 Python 使用 my-uv 管理环境）。


### 3.2 专用智能体列表

#### 3.2.1 基础构建智能体 (`spec-force-base-builder`)
- **定位**: 全能型开发工程师 (Full Stack Engineer)。
- **职责**: 负责通用的代码编写、文档修改、脚本执行及并未被其他专职智能体覆盖的任务。
- **特点**: 灵活性高，具备完整的 "文档<->代码" 双向同步能力。
- **必需的可选模块**:
    - **代码风格**: 必须包含 Python (PEP 8) 和 Markdown 格式规范。
    - **文档结构**: 必须包含通用文档更新规范。
    - **环境要求**: 必须包含完整的 Windows/PowerShell/my-uv 环境配置。
    - **报告规范**: 包含基础执行结果反馈。

#### 3.2.2 需求解析智能体 (`spec-require-explainer`)
- **定位**: 需求分析师 (Requirement Analyst)。
- **职责**: 负责将用户的自然语言需求转化为结构化的 `Specify.md` 任务项，并进一步拆解同步到相关层级规范文档；同时治理 `src/` 文件结构规范与源文件头部“文件自描述”规范，但不直接修改 `src/` 代码文件。
- **核心产出**: 更新后的需求文档与规范文档、结构与文件规范检查清单，不直接修改代码。
- **必需的可选模块**:
    - **文档结构**: **重点**。必须详细定义 `Specify.md`、`Structure.md` 及 Layer/Workflow 文档的模板。

#### 3.2.3 代码实现与运行智能体 (`spec-code-builder`)
- **定位**: 资深工程师 (Senior Engineer)。
- **职责**: 严格依据 `SpecMd` 目录下的规范文档进行代码实现。
- **约束**: **禁止** 擅自增加文档未定义的特性；代码必须与规范 1:1 对应。
- **必需的可选模块**:
    - **代码风格**: **重点**。必须严格定义 PEP 8、Type Hints、Docstrings 及错误处理规范。
    - **文档结构**: 仅需文件头注释规范。
    - **环境要求**: 必须包含完整的运行环境配置以进行简单的自我验证（检查程序是否跑通）。

#### 3.2.4 测试与状态检查智能体 (`spec-tester-and-imple-status-checker`)
- **定位**: 测试工程师 (QA Engineer)。
- **职责**: 编写并运行功能性测试脚本（`pytest` 为主），检查输出产物（Logs, Plots, Files）是否符合 `Specify.md`/相关规范定义的输出范式与验收标准。
- **核心产出**: 测试报告、问题清单、`Specify.md` 中的状态更新。
- **必需的可选模块**:
    - **报告规范**: **重点**。必须定义详细的测试报告格式（概览、失败详情、覆盖率）。
    - **环境要求**: **重点**。必须精确配置测试执行环境（`pytest` + `my-uv`），且环境名优先使用 `$env:PROJECT_ENV_NAME`（若未设置则点源 `SpecMd/scripts/get_env_name.ps1` 写入后使用）。
    - **文档结构**: 涉及 `Specify.md` 状态更新规范。

#### 3.2.5 全局一致性检查智能体 (`spec-whole-consistency-checker`)
- **定位**: 技术审计 (Technical Auditor)。
- **职责**: 静态分析文档与文档之间、文档与代码之间的结构一致性。
- **场景**: 当需求变更后，检查是否有遗漏更新的文档或代码模块。
- **必需的可选模块**:
    - **报告规范**: **重点**。必须定义审计报告格式（问题清单、修正建议）。
    - **文档结构**: 必须理解所有文档类型的结构以进行检查。
    - **代码风格**: 关注代码结构与文档的一致性（如类/函数签名）。

### 3.3 其他智能体

#### 3.3.1 提示词优化智能体 (`prompt-optimizer`)
- **定位**: 提示词工程师 (Prompt Engineer)。
- **职责**: 协助用户将模糊的自然语言需求转化为符合本系统标准架构 (`3.1` 节) 的高质量智能体提示词。
- **核心功能**:
    - **意图分析**: 理解用户希望构建的智能体类型。
    - **架构映射**: 将需求映射到 Metadata, Role, Workflow 等 7 个标准板块。
    - **生成优化**: 输出或直接修改 `agent/spec-*.md` 文件。
- **适用场景**: 创建新的专用智能体或优化现有智能体指令时。

#### 3.3.2 驱动和环境规范智能体 (`driver-env-specialist`)
- **定位**: 环境与脚本专家 (Environment & Script Specialist)。
- **职责**: 处理不完全遵循 "SpecMd 文档驱动" 流程，但必须严格遵守项目环境配置和基础代码规范的任务。
- **特点**: 允许"代码先行"，但严守环境底线。
- **核心约束**:
    - **环境一致性**: 必须使用 `my-uv` 管理依赖，且运行前优先使用 `$env:PROJECT_ENV_NAME`（若未设置则点源 `SpecMd/scripts/get_env_name.ps1` 写入后使用）并激活。
    - **系统兼容性**: 脚本必须适配 Windows PowerShell。
    - **代码规范**: 依然遵循 PEP 8。
- **适用场景**:
    - 编写临时辅助脚本 (Shell/Python)。
    - 环境调试与依赖管理 (`uv add`, `uv sync`)。
    - 快速原型验证 (Prototype) 或实验性代码编写。

### 3.4 智能体使用工作流程

#### 3.4.1 单一智能体模式 (Single Agent Mode)
- **核心智能体**: `spec-force-base-builder`
- **流程**:
    1.  **指令接收**: 用户直接下达自然语言指令（包含需求与实现意图）。
    2.  **自主规划**: 智能体自行决定是否更新文档（通常简化此步）。
    3.  **代码执行**: 直接编写代码并运行验证。
    4.  **结果反馈**: 输出运行结果。
- **适用性**: 简单任务、Bug 快速修复、小功能迭代、临时查询。
- **优点**: 响应速度快，交互成本低。
- **缺点**: 在处理复杂逻辑时，容易导致文档与代码脱节，缺乏系统性保障。

#### 3.4.2 多智能体协作模式 (Multi-Agent Collaboration Mode)
- **核心理念**: 专人专职，流水线作业，通过文档流转控制质量。
- **标准流水线**:
    1.  **需求定义阶段**:
        - 用户输入模糊需求。
        - 调用 `spec-require-explainer`。
        - **产出**: 更新后的 `Specify.md` (明确任务) 及 `SpecMd` 层级文档 (明确设计)。
    2.  **代码实现阶段**:
        - 依据更新后的文档。
        - 调用 `spec-code-builder`。
        - **产出**: 符合规范的代码实现，并确保基础运行无误（无语法/运行时错误）。
    3.  **验证与审计阶段 (穿插执行)**:
        - **功能验证**: 调用 `spec-tester-and-imple-status-checker` 运行测试，生成测试报告，比对 `Specify.md` 验收标准。
        - **结构审计**: 调用 `spec-whole-consistency-checker` 扫描全项目，确保文档与代码结构的一致性。
    4.  **反馈闭环**:
        - 查看上述阶段生成的报告。
        - 若发现问题，重新调用 `spec-require-explainer` (修正设计) 或 `spec-code-builder` (修正实现)。
        - 循环直至所有检查通过。
- **适用性**: 复杂功能开发、大型架构重构、对代码质量和文档完整性有严格要求的任务。
